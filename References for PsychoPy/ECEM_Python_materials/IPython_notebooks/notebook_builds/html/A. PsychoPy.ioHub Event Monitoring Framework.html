High Level ioHub API Review

Starting the ioHub Server Process

There are two ways to create a psychopy experiment and have it start the iohub process for use during the experiment runtime. Both menthods ultimately give you access to an instance of the ioHubConnection Class for communication and control of the iohub framework:

  1. Use the psychopy.iohub.launchHubServer()
  2. Create a subclass of the psychopy.iohub.client.ioHubExperimentRuntime, implementing the classes run() method for your experiment's starting python code.

Each approach has +'s and -'s:

  • Using launchHubServer is quicker when creating an experiment that uses simple devices like the mouse and keyboard.
  • When using more advanced devices like an eye tracker, using the ioHubExperimentRuntime approach quickly becomes easier to manage, as long as you understand the simple file structure expected by the ioHubExperimentRuntime. **

    Starts the ioHub Process using launchHubServer. When started with no key word arguements,

                the ioDataStore will be ective during the experiment. A default name is used if needed.

                to indicate that you are providing a preferences dictionary containing any non default preference

Extending the ioHubExperimentRuntime class

When the ioHubExperimentRuntime class is used to enable the ioHub Process for your experiment, the following experiment file structure must be used:

  • The Python source file containing the ioHubExperimentRuntime subclass for your experiment must be in a directory dedicated to that experiment (lets refer to it as the exp_dir).

  • The exp_dir folder must also contain two configuration files:

    • An experiment_config.yaml file specifying experiment and session level settings.
    • An iohub_config.yaml file that mainly provides the configuration information for each ioHub Device that should be enabled for the experiment.

Devices

In ioHub, a Device can be Physical or Virtual. Physical Devices represent a piece of measurement hardware that generates events. Virtual Devices are purely software based, but can also generate events of interest.

Examples of Physical ioHub Devices:

  • Keyboard
  • Mouse
  • Gamepad
  • Eye Tracker

Examples of Virtual ioHub Devices:

  • Experiment
  • Event Publisher
  • Remote Event Subscriber

A list of all currently supported device types, and associated event types each device can generate, can be found here.

As seen in earlier code, iohub devices are accessed from within the psychopy experiment script via the ioHubConnection class .devices attribute. Notes:

  • The .devices attribute is dynamically created when the experiment starts.
  • The .devices attribute is itself a class, providing dot name access to the iohub devices that were enabled for the experiment. Devices that are not enabled for the experiment will not appear in the devices list.
  • The devices attribute name for a device is equal to the device label specified in the device configuation file (or the default name is used).
  • To programatically check if a device has been enabled from within the experiment script, the deviceByLabel dict attribute of the ioHubConnection class can be used. This is useful so that an experiment can conditionally call a specific device method only if the device is actually enabled.
  • The list of methods that can be called for each device can be accessed within the experiment script.

Notice that the device classes handed to you by ioHubConnection are of type psychopy.iohub.client.ioHubDeviceView. This is the PsychoPy Processes representation of the actual ioHub Device class instances running on the seperate ioHub Server Process.

Notes:

  • For each enabled ioHub Device running during the experiment, their is:

    • One psychopy.iohub.client.ioHubDeviceView instance created by the Experiment Process
    • One iohub.devices.[DeviceClassName] instance created in the ioHub Server Process

  • Any time an the experiment script calls a method of an ioHub Device:

    1. An IPC call is made to the associated ioHub Device on the ioHub Server Process (the request)
    2. Then the ioHub Process sends back the requested information to the Experiment Process (the response)

  • While the IPC request-response delay is very short ( < 0.5 msec on average ), it is still important to keep it in mind. When possible adopt a fetch and reuse approach to reading device values repeatedly within a display refresh cycle. For example:

Events

Each ioHub Device supports 0 - N ioHub Device Events. When a device is being monitored by the ioHub Process, any events that occur:

  • Are converted to a common device type independent event field set + a common device type dependent field set. The event attributes for a given device type are the same regardless of the hardware being used to implement the ioHub Device. This is common sense for devices like the keyboard and mouse (at least within a given OS), but is normally far from the case when using devices like eye trackers or DAQs from different manufacturer's.
  • Are either time stamped by the ioHub process if the event does not have a device time stamp, or has the device time stamp converted to the psychopy / iohub time base.

  • In the case of device time to iohub time base conversion, when possible:

    • Both offset and drift between the two time bases are factored into the conversion.
    • The delay of the event (iohub event received time - device event time) is subtraced from the iohub event time for the device.
  • Are provided to the PsychoPy Experiment Runtime in real-time when requested.

  • Are stored in the ioDataStore; a HDF5 file that uses an event storage structure which constains the same event attributes for each event type as are used by the online accessable event objects during the experiment runtime.

Runtime Event Access:

Events can be retrieved from the ioHub Process from within an experiment script by either:

  • Using the ioHubConnection.getEvents() method.
  • Using the devices getEvent() method.

Events can be cleared from the ioHub Server event buffers using the ioHubConnection.clearEvents() method. Clearing events removes already collected, but not yet requested, events from the ioHub Process buffers so that they not be returned the next time getEvents is called.

Events can be retrived by the experiment process in one of three formats:

  • a tuple of the event attribute values, ordered in the same fashion as the attributes are defined for the event type.
  • a dict, where keys are the event attribute names, and values are ... , well the event attribute values.
  • a namedtuple, allowing access to event attributes as if the evnt was a tuple, or using dot name notation to access the event attributes. i.e. event.time returns the same value as event[TIME_FIELD_INDEX].

By default, events are returned as namedtuples.

ioHubConnection.getEvents()

  • Returns all events from all enabled iohub devices that have been received since the last call to ioHubConnection.getEvents() or ioHubConnection.clearEvents(). So note that calling getEvents() not only returns the events, but removes them from the device type independent event buffer (called the global event buffer).
  • Events are returned sorted by the ioHub time of each event.
  • Events for a specific device type can be requested from the global event buffer by passing the appropriate device name instance string as an arg to the method. In this case, only events from the given device instance are returned and removed from the global buffer.

[device_name].getEvents()

  • Returns events from the device_name device type only, which have been received since the last call to device_name.getEvents() or device_name.clearEvents(). So note that calling device_name.getEvents() not only returns that device's events, but removes them from the device instance dependent event buffer (called a device event buffer).
  • Events are returned sorted by the ioHub time of each event.
  • Events of a specific event type can be requested by providing the desired event type constant to the device_name.getEvents() method.In this case, only events of the requested event type are returned and removed from the device's event buffer.
